The Three Areas
Git tracks your work across three distinct places. Understanding this is the foundation of everything else.
Working Directory
files on disk
git add →← git restore
Staging Area
index / cache
git commit →← git reset
Repository
.git / history
1. You have edited analysis.R but haven't run any Git commands yet. Where does the file currently exist?
AWorking directory only
BStaging area only
CThe repository
DAll three areas simultaneously
✓ Correct. Editing a file only affects your working directory. Git doesn't know about the change until you explicitly stage it with git add.
2. You run git add analysis.R. What has happened?
AThe file is saved to the repository with a commit hash
BA snapshot of the file is copied to the staging area
CThe file is uploaded to the remote repository
DGit starts tracking changes to the file in real time
✓ Correct. git add takes a snapshot of the file at that moment and places it in the staging area. If you edit the file again afterwards, you need to run git add again to include the new changes.
3. Which command moves a snapshot from the staging area into the repository?
Agit push
Bgit add
Cgit commit
Dgit save
✓ Correct. git commit takes everything in the staging area and permanently stores it as a new commit in the local repository. git push comes later — it sends commits to the remote.

Reading git status
Being able to read status output quickly is one of the most important habits you can build.
4. What does this output tell you?
On branch main Your branch is ahead of 'origin/main' by 2 commits. (use "git push" to publish your local commits) Changes not staged for commit: modified: report.Rmd
AEverything is up to date — nothing to do
Breport.Rmd has been staged and is ready to commit
CThere are 2 new commits on the remote that you need to pull
DYou have 2 unpushed local commits, and report.Rmd is modified but not staged
✓ Correct. "Ahead by 2 commits" means you have 2 local commits the remote doesn't have yet — run git push. The modified report.Rmd is separate: it's changed on disk but not staged yet.
5. You see this output. What is the state of data.csv?
On branch main Changes to be committed: new file: data.csv Changes not staged for commit: modified: README.md
AIt is a new file that Git has not seen before — untracked
BIt is staged and will be included in the next commit
CIt has been committed to the repository
DIt is modified but not staged
✓ Correct. "Changes to be committed" means the file is in the staging area — it will be part of the next git commit. Note README.md is in a different state: modified on disk but not yet staged.

Scenario Questions
Read the terminal output and answer the question about what to do next.
12. Your colleague pushed new commits. You run git status. What should you do to get their changes?
$ git status On branch main Your branch is behind 'origin/main' by 3 commits, and can be fast-forwarded. (use "git pull" to update your local branch) nothing to commit, working tree clean
Agit push origin main — push your branch to sync it
Bgit pull origin main — fetch and merge the remote commits
Cgit fetch origin — this will also update your local branch
DNothing — "nothing to commit" means everything is fine
✓ Correct. "Behind by 3 commits" means the remote has commits you don't have locally. git pull fetches and merges them. Option C (git fetch) would only download — it would not update your local branch.
13. You staged a file by mistake. Which command removes it from the staging area without deleting it from disk?
$ git status Changes to be committed: new file: secret_passwords.txt <-- you don't want this committed
Agit reset --hard HEAD
Bgit checkout -- secret_passwords.txt
Cgit restore --staged secret_passwords.txt
Dgit rm secret_passwords.txt
✓ Correct. git restore --staged removes the file from the staging area while leaving it untouched on disk. Option A discards all uncommitted changes. Option D deletes the file entirely. Option B discards working directory changes.
14. What does the (HEAD -> main, origin/main) annotation tell you about this commit?
$ git log --oneline a3f92cd (HEAD -> main, origin/main) Add data cleaning script 7b1e034 Initial commit
AYour local main and origin/main both point to the same commit — you are in sync
BThere is a conflict between your local branch and the remote
CHEAD is detached and pointing directly to a commit
DThe commit exists on the remote but not locally yet
✓ Correct. When both HEAD -> main and origin/main point to the same commit, your local branch and the remote are in sync. If you make a new commit, only main will advance and origin/main will fall behind — until you push.
